## 2. Решения до старта

- **D1 — РЕШЕНО: вариант (а), де-ранжирование (2026-07-10, Игорь делегировал: «проверь историю,
  если логично — делай»).** Проверка истории показала: это НЕ новое решение, а восстановление
  четырёх уже залоченных. (1) P4-lock + DECISIONS.md D3: «first DOI = pilot + coverage-map,
  **not a founding leaderboard** … not a rankings table»; (2) COST-TO-ATTACK: **no-rank =
  security-фича** (ограничивает приз атакующему/геймящему); (3) CONSTITUTION: «lets the reader
  **sort for themselves**»; (4) консилиум-26 (observation≠assessment): авторские вердикты — только
  ≥Rising-порога, «стороннюю сортировку CC0-фактов ты не авторствуешь — режим OWID»; плюс URL
  `/rankings/*` уже осознанно депрекированы в `/ai/cohorts/*`. Порядковый RankTable и «ranked by
  momentum» в llms.txt — дрейф против всех четырёх слоёв. Исполнение: WP-E (гейт снят).
  Momentum-значение у каждой системы ОСТАЁТСЯ (это измерение); убираются порядковые номера и
  дефолт-лестница; клиентский toggle сортировки допустим.
- **D2 (решено оркестратором):** URN-эпоха = `snapshot_date` (не `period`); `snapshot_id` =
  content-hash канонического снапшота. Старые URN не переминчиваются — errata-маппинг. Основание:
  Codex-находка F2 подтверждена данными; `snapshot_date` уже поддержан грамматикой CLAIM-URN.
- **D3 (решено, механизм — на план-гейте WP-D):** person-slugs заменяются на публичных поверхностях;
  `github_repo` как ВНУТРЕННИЙ natural key минтинга ID НЕ трогаем (иначе осиротеет история — латентный
  риск из аудита, Gemini). Уже опубликованные утечки в append-only снапшотах — через privacy-tombstone
  (конституция это явно разрешает как единственный легитимный путь).

## 3. Глобальные правила исполнения (для каждого WP)

1. Чистое дерево до делегации; после — дифф-скан скоуп-забора; вне забора → откат.
2. **Frozen-etl инвариант:** `etl/collect.py` НЕ редактируется. Всё чинится пост-шагами вне frozen
   (паттерн уже в репо: `collectors/gen_redirects.py`, `taxonomy_v2.py` restore). Codex обязан
   соблюдать; ревьюеры проверяют.
3. Append-only: опубликованные снапшоты не переписываются. Исправления = новый снапшот + ERRATA-слой.
   Единственное исключение — privacy-tombstone (D3).
4. Тесты, кто гоняет:
   - Codex — юнит-тесты локально в ходе работы (в спеке каждого WP — команды);
   - **Claude на приёмке — всё сам:** `cd web && npm run verify` (vitest+build+check-dist),
     `.venv/bin/python -m pytest` (корень), затем push и `gh run list/watch` до зелёного CI;
   - Grok — post-deploy живые проверки (перечислены в WP);
   - Gemini — финальный 1M-реаудит (§6).
5. Ревью — только голый дифф (`codex-review`/`gemini-review` строят сами), VERDICT cap 3.
6. Каждый WP закрывается: ledger-оценка (`codex-rate`/`gemini-rate`), коммит `bin/sb-save`-стилем
   в репо Evidaxis (обычный git commit + push, репо своё), строка в этом файле: `[x] WP-… done <дата>`.
